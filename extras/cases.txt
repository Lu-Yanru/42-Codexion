Guaranteed burnout:
- 1 coder e.g. ./codexion 1 800 200 100 200 5 0 fifo
- time_to_burnout < time_to_compile e.g. ./codexion 4 100 200 100 200 5 0 fifo
- time_to_burnout < time_to_compile + time_to_debug + time_to_refactor e.g. ./codexion 4 300 200 100 200 5 0 fifo
- For even number coders: time_to_burnout < (time_to_compile + time_to_cooldown) * 2 e.g. ./codexion 4 500 200 100 100 5 60 fifo
- For odd number coders: time_to_burnout < (time_to_compile + time_to_cooldown) * 3 e.g. ./codexion 5 800 200 200 200 5 80 fifo

Guaranteed success:
- number_of_compiles_required is set to 0 (nothing happens) e.g. ./codexion 1 800 200 100 200 0 0 fifo
- For fifo: time_to_burnout > (time_to_compile + time_to_cooldown) * number_of_coders e.g. ./codexion 4 1210 200 100 100 5 100 fifo
- For edf: time_to_burnout > (time_to_compile + time_to_cooldown) * 3 or 2 e.g. ./codexion 5 1810 200 100 100 5 400 edf     ./codexion 4 450 140 20 20 3 60 edf

Succeed on edf but not on fifo:
- ./codexion 4 800 200 200 200 5 10 fifo
Sometimes burned out because of deadlock at start of the program (ddl the same, queue order solely depend on time of entry).
e.g. coder 1 queued for dongle 1 and dongle 0 first, then coder 2 queued for dongle 1 and dongle2,
then coder 3 queued for dongle 3 and dongle 2, then coder 4 queued for dongle 3 and dongle 0.
In this case, under fifo, coders can only compile one by one, resulting in burn out.
-> Implemented tie-breaker in edf (ddl the same at start of the program so can be tie-broken)

- ./codexion 4 800 0 0 0 5 200 fifo
May burnout because coders that finished compiling may queue immediately and may be faster than the coders that didn't compile.
But edf should always make the waiting coders compile first.
-> problem: 1 enqueued and dequeued so fast that 2 did not have time to yield
